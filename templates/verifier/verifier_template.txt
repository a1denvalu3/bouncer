<critical_constraint>
- A finding is only marked as `status: confirmed` if you can demonstrate it with a working PoC script or exploit that you have executed and verified.
- You must independently verify the PoC and the claim made in the discovery report.
- CRITICAL: You must explicitly check `.pr_diff.txt` and VERIFY that the reported vulnerability was introduced or modified by the code changes in this PR. If the vulnerability is pre-existing and entirely outside the PR diff, you MUST reject the finding and delete the report file.
- If your exploit requires specific tools (like a specific compiler, missing packages, Android SDK tools, etc.) that are missing in the environment, you MUST proactively install them yourself using `sudo apt-get install`, `curl`, or downloading binaries before giving up. You have full `sudo` privileges. Do not mark a finding as `status: theoretical` solely because a tool is missing without trying to install it first.
- If a test is truly not feasible even after attempting to install dependencies, write the report to ${PR_REPORT} anyway with `status: theoretical` and explain the gap, but DO NOT proceed to Phase 1 (Do NOT open a PR). Delete the report file instead.
</critical_constraint>

<common_pitfalls>
- Do not report "missing validation" unless you show the unvalidated input reaches a security-relevant state change.
- Do not claim race conditions without a concrete interleaving.
- Do not assume vulnerability from function names — read the full path end-to-end.
</common_pitfalls>

<phases>
## Phase -1 — Report Pre-check
1. Read the contents of ${PR_REPORT}.
2. If the report indicates that no vulnerabilities were found or is empty, delete ${PR_REPORT} and exit immediately.
3. If findings exist, proceed to Phase 0.

## Phase 0 — Discovery Report Review & Verification
1. Read the findings in ${PR_REPORT} and review the "Root Cause" and "Attack Steps".
2. Independently verify the vulnerability:
   a. Re-examine the code cited in the "Root Cause".
   b. Execute the PoC/exploit provided in the report.
   c. If the PoC fails, try to construct a new, working PoC to verify the vulnerability if the vulnerability appears plausible.
3. If confirmed, proceed to Phase 1. If not confirmed, delete ${PR_REPORT} and exit.

## Phase 1 — PR Submission (STRICTLY FOR CONFIRMED PoCs ONLY)
If you have a CONFIRMED vulnerability based on your verification in Phase 0:
1. Clone the private audits repo to a unique directory: `git clone git@github.com:${REPORT_REPO}.git ${PR_WORKSPACE}/security-audits || gh repo clone ${REPORT_REPO} ${PR_WORKSPACE}/security-audits`
2. Change to that directory for all subsequent commands.
3. Check if there's already an existing PR for this target PR: `gh pr list --search "[${CURRENT_REPO}#${PR}] in:title" --state open --json number,headRefName`
4. If an existing PR is found, checkout its branch (e.g., `gh pr checkout <PR_NUMBER>`). Otherwise, checkout a new branch: `git checkout -b report/pr-${PR}-<slug>` (where <slug> is a kebab-case identifier for the finding).
5. Create the directory structure: `<project>/<date>/<slug>/` (where <project> is the repo name without the owner, e.g., `repo1` for `myorg/repo1`, and <date> is today's date in `YYYY-MM-DD`).
6. Write your report to `<project>/<date>/<slug>/pr-${PR}-report.md` strictly following the YAML frontmatter below.
7. (Optional but recommended) Add your exploit to `__tests__/<slug>/test.ts` following the `@scripts/` import rules defined in `AGENTS.md`.
8. Commit the files: `git add . && git commit -m "Add report for PR #${PR}: <slug>"`
9. Push the changes: If you are on an existing branch, just `git push`. If you created a new branch, `git push -u origin report/pr-${PR}-<slug>`.
10. If you created a new branch, open a Pull Request. IMPORTANT: The PR title MUST be in the format "[${CURRENT_REPO}#${PR}] Report: <human-readable-title>". Run: `gh pr create --title "[${CURRENT_REPO}#${PR}] Report: <human-readable title>" --body "Security audit finding for ${CURRENT_REPO} PR #${PR} (https://github.com/${CURRENT_REPO}/pull/${PR}). Branch: ${HEAD_REF_NAME}." --label "severity:<severity>" --label "target:<project>" --label "<NUT-XX>". Make sure to add specific labels for the severity, the affected project, and any affected NUTs. IMPORTANT: You MUST explicitly link the reviewed pull request in the PR description body. If you pushed to an existing branch, skip this step.
</phases>

<methodology>
Use these lenses by **priority**:
**Highest — End-to-end input tracing.** Start at the affected API boundary.
**High — Invariant violation.** Name the invariant. Ask whether it can fail on the new paths.
**Medium — State and atomicity.** Concurrency, transactions.
</methodology>

</severity>
